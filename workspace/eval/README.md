# Eval 量化框架

> Peter 原则：不量化就不知道自己变好了还是变差了。
> 对高频任务建 baseline，每次 prompt/流程变更后跑 eval，用数字说话。

## 原则

1. **先有 baseline 再改** — 改 prompt 前先跑一次 eval 存档，改后再跑，对比
2. **评估维度标准化** — 每类任务用固定维度打分（1-5），不主观飘
3. **sample set 固定** — 用同样的输入跑不同版本的 prompt，才有可比性
4. **自动化 > 人工** — 用第二模型当 judge，减少人工审核负担

## 评估维度

### 内容类（翻译/摘要/推文）
| 维度 | 权重 | 1 分 | 5 分 |
|------|------|------|------|
| 准确性 | 30% | 事实错误/误译 | 零错误 |
| 可读性 | 25% | 生硬/机翻感 | 自然流畅 |
| 风格匹配 | 20% | 完全不像目标风格 | 完美匹配 |
| 信息密度 | 15% | 废话多/遗漏关键 | 精简完整 |
| AI 腔 | 10% | 满屏 AI 套话 | 零 AI 感 |

### Cron 产出类（情报/监控/运维）
| 维度 | 权重 | 1 分 | 5 分 |
|------|------|------|------|
| 信噪比 | 35% | 全是噪音 | 每条都有价值 |
| 行动性 | 25% | 纯信息堆砌 | 直接可执行 |
| 准确性 | 20% | 信息错误 | 全部准确 |
| 格式规范 | 10% | 格式混乱 | 完美标准化 |
| 及时性 | 10% | 过期信息 | 实时/当日 |

### 代码类
| 维度 | 权重 | 1 分 | 5 分 |
|------|------|------|------|
| 正确性 | 40% | 不能运行 | 测试全过 |
| 可维护性 | 20% | 无法理解 | 清晰自解释 |
| 最小改动 | 20% | 改了一堆不相关的 | 精准修改 |
| 安全性 | 10% | 有明显安全问题 | 无问题 |
| 性能 | 10% | 明显低效 | 合理高效 |

## 目录结构

```
eval/
├── README.md           # 本文件
├── samples/            # 固定样本集
│   ├── translation/    # 翻译样本（原文 + 参考译文）
│   ├── cron-output/    # Cron 产出样本（prompt + 期望产出）
│   └── summary/        # 摘要样本
├── baselines/          # 基线分数存档
│   └── YYYY-MM-DD.json # 每次 eval 的完整打分
├── judge-prompt.md     # 给 judge 模型的评分指令
└── run-eval.sh         # 自动化评估脚本
```

## 工作流

### 建 Baseline
1. 收集 5-10 个代表性样本到 `eval/samples/<类别>/`
2. 用当前 prompt 跑一遍，存产出
3. 用 judge 模型打分，存到 `eval/baselines/YYYY-MM-DD.json`

### 改 Prompt 后验证
1. 用**同样的样本集**跑新 prompt
2. 用**同样的 judge prompt**打分
3. 对比 baseline：总分提升 → 合并；总分下降 → 回滚或继续调

### 定期回归
每周系统进化 review 时，抽查 2-3 类任务跑 eval，防止退化。
